# Hazync — soundness audit, 2026-07-22 (round 8)

An adversarial audit of whether a Hazync proof *actually validates a Bitcoin block fully*, followed by
fixes for every issue found. Conducted with five independent reviewers, each given a single dimension
and an explicit mandate to break it. This document records what was checked, what was found, what was
changed, and what a verifier/deployer must do next. Like the rest of `SECURITY.md`, it is written to
surface critique, not hide it.

The headline: the validation **logic** is sound and consensus-complete for the reachable chain — it runs
real Bitcoin Core v28 code, and the classic zkVM-prover soundness holes (host-chosen height / flags /
prevouts / wtxids) are all closed. Five deviations from Core's block-connection rule set were found;
none allowed minting, theft, or double-spend on the real chain today. All are now fixed or (for the one
unprovable rule) documented.

## Scope & method

Reviewed at commit prior to this round: `prover/methods/guest/src/main.rs` (the zkVM guest — all proving
modes), `prover/methods/guest/verify_input.cpp` (the C++ FFI: real Core `VerifyScript`, sighash,
`CheckTransaction`, PoW, retarget, merkle, witness commitment, sigops, subsidy, locks), `build.rs` (which
Core TUs compile into the guest), `prover/methods/guest/src/utreexo.rs` + `accumulator/src/lib.rs` (the
UTXO accumulator), and `prover/host/src/main.rs` (witness construction, the anchor, the verifier
commands).

Five reviewers, one dimension each, each told to assume a malicious prover controlling every witness byte:

1. **Consensus completeness** — is any Core mainnet block-connection rule *missing* (accept-invalid)?
2. **Enforcement gating** — is every computed check actually on an asserted panic path (no proof for an
   invalid block)?
3. **UTXO accumulator soundness** — inclusion forgery / skipped-delete / double-spend?
4. **FFI / VerifyScript boundary** — does the real script run against the *authenticated* coin, with the
   correct amount / sighash / flags?
5. **Trust scope & verifier** — where does the proof bottom out (genesis vs checkpoint), and does the
   top-level verifier tie everything to genesis?

## Verdict by dimension

| Dimension | Verdict |
|---|---|
| UTXO accumulator | **Sound.** Delete is location-pinned (SEC-2); leaf-binding is second-preimage-tight with unambiguous field boundaries; single-spend enforced by accumulator-shrink (external) and the `created_at`/`spent_in_block`/strictly-earlier-tx rules (in-block); the resulting root is guest-recomputed and carried across every chain/range/fold/aggregate seam. No forging witness found. |
| FFI / VerifyScript | **Sound.** Script, BIP143/BIP341 sighash amount, and accumulator leaf all read the *same* `CTxOut`; the full prevout set is force-computed into `PrecomputedTransactionData` with `MissingDataBehavior::FAIL`; flags are guest-derived and consensus-exact; return/trap discipline is strictly fail-closed. |
| Enforcement gating | **Sound.** Every consensus check in the proving modes (2 `chain_step`, 5 `aggregate`, 6 `prove_range`, 7 `fold_range`) is in the asserted conjunction. Mode 1 (`block_proof`) is a non-asserting report path, confirmed debug-only and un-launderable via the `KIND_*` domain tags. **One finding:** the `is_base` anchor was not committed into `ChainState` (see A1). |
| Consensus completeness | **Complete for the reachable chain**, with the deviations below (G1–G5). Every economically meaningful rule is enforced with real Core code and the comments survive verification. |
| Trust scope | Production **range-fold** path bottoms out at **true genesis** (height 0, empty UTXO set) — no assumed prefix. Real STARK proofs currently exist only to ~block 3,050; beyond that the pipeline is logic-validated in execute mode. See "What is proven vs assumed". |

## Findings & fixes

| ID | Severity | Finding | Fix |
|----|----------|---------|-----|
| **A1** | verifier-hole | A bare `ChainState` (mode-2/5) receipt did not commit its anchor, so a receipt built on a *fabricated* anchor (arbitrary height/UTXO/work/easy nBits) is journal-indistinguishable from a genesis-anchored one. Not exploitable through any shipped verifier (`verify-range`/`verify-any` accept only `KIND_RANGE` and pin genesis), but a standing foot-gun. | **S5:** commit `anchor_id = dsha256(base-anchor journal)` into `ChainState`, set once in the `is_base==1` step and carried forward unchanged; new host command **`verify-chain`** pins it to `dsha256(genesis_anchor)` — the chain-track analogue of `verify-range`'s genesis in-boundary pin. |
| **G3** | medium (reject-valid / liveness) | The BIP141 witness-commitment check ran at **all** heights; Core gates it on segwit-active (≥ 481824). Canonical pre-activation blocks that carry an early commitment output but a witness-free coinbase would be wrongly *rejected* — a from-genesis stall in the 433k–481823 range. | Enforce the commitment only from height 481824; below activation, `witness_ok = !has_witness` (Core's `unexpected-witness`). |
| **G2** | low | `unexpected-witness` excluded the coinbase from `has_witness`; Core's loop runs over *every* tx including the coinbase. | `has_witness` now includes the coinbase's own witness (the reserved-value nonce). |
| **G5** | low | Block-level `MoneyRange(nFees)` was implicit (each per-tx fee was range-bounded, but the block total was not asserted, and `subsidy + total_fee` was an unguarded i64 add). | Explicit `MoneyRange(total_fee)` + i128-safe `subsidy + fees ≤ MAX_MONEY`, folded into the already-gated `subsidy_ok`. |
| **G1** | low (bounded) | The general BIP30 `HaveCoin` check (no created outpoint may duplicate an existing unspent coin) is replaced by a structural argument, not a lookup — a utreexo `Stump` cannot prove non-membership. | Made the argument an **explicit, gated, bounded invariant** in-code: BIP34 (asserted, ≥ 227931) forces coinbase-txid uniqueness; the two pre-BIP34 duplicates (91842/91880) are handled by the F3 overwrite; a non-coinbase duplicate is a double-spend the accumulator rejects. Sound for every mainnet block up to the first BIP34/BIP30 collision height (~1,983,702, ≈ 2046), where a real membership/overwrite mechanism (utreexo non-membership) would be needed — now stated in-code, not assumed. |
| **G4** | (not a bug) | The 2-hour future-timestamp limit is not enforced. | **Intentionally left.** It depends on the verifier's local wall-clock — mathematically unprovable in a zkVM. Kept documented as a trust boundary. |
| **N1** | hygiene | Dead `BlockInput.flags` wire field (never read; flags are guest-derived) — a refactor trap that a future change could start honouring. | Removed from guest + host + all construction sites (and the now-dead `Spend.flags` host helper field), with a comment warning against re-adding a host flags input. |
| **N2** | fragility | The `scriptPubKey` in the UTXO-leaf commitment was not length-prefixed — injective today only because it is the sole variable-length field; adding a second would create ambiguity. | Length-prefixed `scriptPubKey` in all three byte-identical leaf sites (`coin_leaf` + `tx_out_leaves` in C++, `coin_leaf` in the host). **Changes every leaf hash / accumulator root** — see re-derive checklist. |
| **N3** | fragility | `tx_full_sigops` returned the legacy-only sigop cost on a short (malformed) prevouts blob — safe only because `verify_input` independently rejects the block. | Fails closed: returns a poison sigop cost (`1 << 40`, above the 80k limit but overflow-safe under summation) so `sigops_ok` becomes false directly. |

## What is proven vs assumed (the answer to "does it validate a block fully?")

**Proven** (in-guest, real Core code, per block from the anchor): script validity of every type with the
correct per-height flags (incl. the two mainnet flag-exception blocks); `CheckTransaction`; no-inflation
(fees + exact subsidy schedule, now with block-level `MoneyRange`); PoW + retarget; merkle root incl. the
CVE-2012-2459 mutation; BIP141 witness commitment (now activation-gated) + `unexpected-witness`; BIP34;
BIP30 (bounded invariant, incl. the two grandfathered overwrites); weight ≤ 4M; full sigop cost ≤ 80k;
coinbase maturity; absolute + BIP68 relative locktime (height and real-MTP time); `time-too-old`;
prev-hash linkage; the UTXO transition through the Utreexo accumulator; cumulative work; recursion
image-id binding; and now the **anchor identity** (S5).

**Assumed**: the anchor state itself (genesis constants for the production range-fold path — negligible;
a checkpoint's whole history for the demo paths); SHA-256 collision resistance; RISC0 STARK soundness;
that the committed `cum_work` is the *most-work* chain (an external comparison the verifier still makes);
and the 2-hour future-time rule (G4, unprovable).

**Scope today**: a verified **range-fold** receipt (`verify-range`) attests full validity from **genesis
(empty UTXO set)** through its `hi` — no assumed prefix. Such receipts currently exist to ~block 3,050
(the frozen board frontier; 550 in committed evidence). Beyond that the identical guest logic is
execute-validated on real blocks but not yet cryptographically proven; the genesis→tip "Proof Party" is
the open scaling item, not a soundness defect. Standalone single-block receipts (e.g. 741000) prove
*internal* validity only, not real-UTXO membership.

## Verification of the fixes

All changes are consensus-neutral on valid data and correctly reject invalid data — verified in execute
mode (real Core code in the zkVM, no proving):

- **Build**: guest (Core → RISC-V) + host build clean; `cargo fmt` clean.
- **Demo 170→172**: tip hashes byte-identical (`eea2d48d…`, `e0b4bf8d…`, `54686892…`).
- **`check-full` 130000** (legacy, pre-segwit G3 branch): **VALID**, tip `211c58f6…`, matches the
  pre-change recorded value.
- **`check-full` 741000** (segwit + taproot, active G3 branch, ~670 inputs): **VALID**, tip `f0e89160…`,
  394 UTXO leaves — matches the pre-change recorded value. Because the tips are header-derived, their
  byte-for-byte match proves the N2 leaf-format change is consensus-neutral (the accumulator roots change
  on both the guest and host sides consistently, so `root_matches` still holds).
- **`check-full` 741000_badwit** (witness negative test): **REJECTED**, `witness_ok=false` only, every
  other flag true — clean isolation after the G2/G3 rewrite.
- **`check-bip30`** (F3 grandfathered overwrite, block 91842): PASS — honest overwrite accepted with a
  matching root; skipped-overwrite and wrong-old-height both rejected (validates N2 on the BIP30
  coinbase-leaf-recompute path).
- **`regress`**: PASS — block 170 `chain_step` tip match + `self_id` OK.

Not yet run (needs the GPU box): `clippy` / `cargo check --all-targets`, the prove-based `adversarial`
S1/H2 suite, and one real STARK prove → **`verify-chain`** end-to-end (confirm `anchor_id` pins to
genesis; note the demo/checkpoint proofs, anchored at 169, *should* fail `verify-chain` — that is the fix
working, not a regression).

## Reproducibility & METHOD_ID

The reproducible-build **mechanism** is unchanged: the pinned toolchain (rzup rust 1.94.1 / cpp 2024.1.5 /
cargo-risczero + r0vm 3.0.5), risc0 crates `=3.0.5`, Bitcoin Core v28.0, secp256k1 v0.5.1, committed
`Cargo.lock`, fixed build paths, `reproduce/Dockerfile` + `provision-vps.sh` — **none of these changed**.
Only guest/host source changed (the fixes), plus cosmetic `cargo fmt` reformatting of `build.rs` /
`utreexo.rs` (whitespace only; zero effect on the compiled guest ELF).

The **METHOD_ID value changes** — necessarily, because it is a content hash of the guest program and the
guest program changed (it also changes because of the N2 leaf-format change). It changes to one new
deterministic value that everyone building the pinned source gets identically; reproducibility is
preserved, the id is just re-pinned. The canonical value is regenerated by
`docker build -f reproduce/Dockerfile .` and committed to `reproduce/METHOD_ID`; all published proofs
must be re-proven against it.

## Re-derive / re-prove checklist (before deploy)

1. `docker build -f reproduce/Dockerfile .` → capture the new canonical `METHOD_ID`; update
   `reproduce/METHOD_ID` (supersede `c029cee4…`) and any doc that cites it.
2. Run `clippy` / `cargo check --all-targets` and the prove-based `adversarial` suite on the box.
3. Re-prove the board from genesis on the new guest; the old proofs are invalid (METHOD_ID + leaf format
   changed).
4. Prove one small range and confirm `verify-range` and (for a chain receipt) the new `verify-chain`
   both pass, and that a checkpoint-anchored proof correctly fails `verify-chain`.
5. Rebuild + sign the host release; deploy the coordinator with the new host binary and re-point the board.
